Gangent 流程图总览

01 展示完整 Runtime 闭环；后续每页沿同一立式状态流拆开一个关键模块，便于分别理解输入、控制、执行、异常分支与反馈闭环。

00 / Index

01 总体运行闭环

主链路始终从上到下：先建立任务和计划，再进入模型决策与工具执行，最后更新状态、验证结果并选择完成或恢复。左右分支只表示 Fast Path、Tool Call 和异常处理。

01 / Runtime

02 Planner + Budget + Model Routing

Planner 生成候选计划，Budget 在规划阶段分配资源边界，Model Routing 为步骤选择合适模型；Runtime 仍负责实际扣减和硬限制。反馈数据回到下一次规划，而不是无限增加 Step 或 Token。

02 / Planning

03 Tool Calling + Policy

模型只能提出工具动作。Tool Boundary 先检查工具、参数和当前步骤权限，Policy 再决定允许、拒绝或人工审批；只有通过门禁后 Tool Runtime 才能产生真实副作用。

03 / Tool Boundary

04 Validator + Finish Guard

Validator 执行具体检查，Finish Guard 根据检查结果决定能否结束。模型只能请求 Finish，不能凭一句“已完成”把任务直接标记为 completed。

04 / Completion

05 Checkpoint + Resume + Error Recovery

Checkpoint 保存可恢复事实，Resume Health Check 判断旧状态是否仍可用，Error Recovery 根据错误类型、已有副作用和剩余计划选择有限重试、修复、重规划、等待用户或失败。

05 / Recovery

06 Dynamic Context + Memory

Dynamic Context Loading（动态上下文加载）按当前任务和步骤选择 ContextSegment，并在 Token 预算内做过滤、压缩和溯源；Memory 只是候选来源之一，不会把全部历史直接塞进 Prompt。

06 / Context

07 Event Queue + Replanning

运行中输入先标准化为事件并排队，不强行杀死正在进行的模型调用。Runtime 到达 Safe Boundary 后，把原始输入、历史补充、当前状态和阶段成果打包成 ReplanContext，再决定继续、追加、替换、暂停或稳定化。

07 / Event Runtime

08 Log + Audit + Evaluation

Log 面向系统调试与监控，Audit Trail（审计轨迹）面向业务追责与证据链，Evaluation 使用任务结果和指标判断系统是否真的变好；三者共享事件来源，但保留不同粒度和访问边界。

08 / Observability

09 CLI / Web Shell（可选）

CLI 和 Web Shell 是可替换的交互适配层。CLI 提供最稳定的同步入口；Web Shell 在后台运行任务，展示 Activity、Checkpoint、Event 和 Token 状态，并把运行中的补充输入写入事件队列。

09 / Interface
